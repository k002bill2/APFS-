# 🌐 웹 접근성(Web Accessibility, A11y) 가이드 — APFS

이 문서는 모든 사용자(장애인·고령자 포함)가 동등하게 APFS 대시보드를 이용할 수 있도록 하는 **원칙 원문**이다. 실제 구현 규약·정본 패턴·검증 절차는 **`web-a11y` 스킬**(`.claude/skills/web-a11y/SKILL.md`)에 있고, 이 문서는 그 배경이 되는 4대 원칙과 체크리스트를 다룬다.

> **APFS 맥락 주의** — 이 프로젝트는 **React 인라인 `style` + 스키마 주도 폼 + shadcn/Radix 프리미티브** 구성이다. 따라서 아래 원칙을 정적 HTML(`<form><label for>`, `<img alt>`)로 직역하지 말고, 코드베이스가 세운 패턴(`IconBtn`·`Field`·`RowFormModal`·`DatePicker`)으로 구현한다. 각 원칙 아래 **▶ APFS 적용**에 매핑을 적었다.

근거 기준: **KWCAG(한국형 웹 콘텐츠 접근성 지침)** — WCAG 2.1 기반, 4대 원칙(인식·운용·이해·견고). 국내 표준은 KWCAG 2.1(4원칙·13지침·24검사항목)이 널리 쓰이고, 개정판 KWCAG 2.2도 공개되어 있다(항목 수 상이) — 실무는 4대 원칙 준수가 핵심.

---

## 📌 4대 원칙

### 1. 인식의 용이성 (Perceivable)
모든 콘텐츠는 사용자가 인식할 수 있어야 한다.
- **대체 텍스트**: 의미 있는 이미지에 `alt`, 장식 이미지는 빈 값 `alt=""`.
- **자막**: 동영상/오디오에 자막·수화(현재 APFS엔 미디어 없음 — 추가 시 적용).
- **명도 대비**: 텍스트:배경 **4.5:1 이상**(대형 텍스트·비텍스트 UI는 3:1).
- **콘텐츠 선형화**: CSS에 의존하지 않는 논리적 DOM 순서.
- **색만으로 정보 전달 금지**: 색+텍스트/아이콘 병행(색맹 대응).

**▶ APFS 적용**
- 색·대비·다크모드 전경은 **`color-tokens` 스킬 / `docs/COLOR_TOKENS.md`** 소관 — hex 하드코딩 금지, `tokens.css` 변수만. 라이트/다크 **양 테마 모두** 대비 확인.
- 자체 SVG 차트(`charts.tsx`)는 SR이 못 읽으므로 인접 텍스트/표로 대체 정보 제공.
- 로고 `<img>`엔 이미 `alt` 부여됨(shell.tsx). 데이터 마스크(`MT`/`mn`)는 축·헤더·제목을 가리지 않는다("축은 두고 데이터만 가린다").
- StatusBadge 등 상태는 tone(색)만이 아니라 라벨 텍스트 병행.

### 2. 운용의 용이성 (Operable)
UI 구성요소는 조작 가능해야 한다.
- **키보드 접근성**: 마우스 없이 **모든 기능**을 Tab/Enter/Space/Esc/화살표로 조작.
- **초점 이동**: 초점 순서가 논리적이고, 현재 초점이 **시각적으로 보임**.
- **충분한 시간**: 시간제한·자동 이동이 있으면 정지·연장 제공.
- **터치 타깃 크기**: 핵심 조작요소 충분한 크기.
- **깜빡임/움직임**: 자동재생·번쩍임 억제, 모션 축소 존중.

**▶ APFS 적용**
- 초점 링은 `tokens.css`의 전역 `:focus-visible{outline:3px …--ring…}` — **`outline:none`으로 지우지 말 것**.
- 모션 축소는 `tokens.css`의 전역 `@media (prefers-reduced-motion:reduce)`가 CSS 애니메이션을 kill. **JS 기반 자동이동(캐러셀·자동 페이지네이션)은 직접 `matchMedia`로 가드**(전역 규칙이 못 잡음).
- 커스텀 드롭다운/플라이아웃/탭은 화살표+Esc 키 지원 + `role`/`aria-expanded`, 또는 **Radix 프리미티브를 그대로 사용**(키보드·초점 자동).
- 터치 타깃 24px 규약은 **`responsive-ui` 스킬** 소관.

### 3. 이해의 용이성 (Understandable)
콘텐츠는 이해하기 쉬워야 한다.
- **가독성**: 쉬운 언어, 축약어·전문용어 풀이.
- **예측 가능성**: 새 창·자동 소리 등 예상 못 한 동작을 갑자기 실행하지 않음.
- **일관성**: 내비게이션·메뉴 구조를 전 페이지 일관 유지.
- **입력 도움(오류 처리)**: 오류의 **위치와 해결 방법**을 텍스트로 안내(색만으로 X), 오류 필드로 초점 이동.

**▶ APFS 적용**
- 폼 오류는 `Field`의 `errMsg`로 필드별 텍스트 표시. 색상 테두리만으로 오류를 나타내지 않는다.
- GNB/LNB 3-레벨 구조와 브레드크럼은 `Shell`이 전 페이지 일관 렌더(RBAC로 항목만 가시성 분기).
- 외부 링크·새 창은 예고(`aria-label`에 "새 창" 등, 예: asset_funding.tsx).

### 4. 견고성 (Robust)
미래 기술·보조기기와 호환되어야 한다.
- **웹 표준 준수**: 유효한 마크업, 중복 `id` 없음.
- **호환성**: 다양한 OS·브라우저·스크린리더와 동작.
- **이름·역할·값(Name/Role/Value)**: 커스텀 컨트롤도 역할·상태를 보조기기에 노출.
- **상태 메시지(Status Messages, WCAG 2.2 4.1.3)**: 초점 이동 없이도 동적 변화를 알림.

**▶ APFS 적용**
- 클릭 요소는 `<button>`, 내비는 `<nav>`, 제목은 `<h1>~<h6>` — `<div onClick>`로 흉내 금지(역할·포커스·엔터 손실).
- 아이콘 전용 버튼은 **`IconBtn`의 `label`** → `aria-label`(+Tooltip). 토글은 `expanded` prop → `aria-expanded`.
- **상태 메시지 = ARIA 라이브리전**: 조회완료·저장·에러·로딩을 `Alert`(`role="alert"`)·`Skeleton`(`aria-live="polite"`)·`Spinner`(`role="status"`)로 SR에 전달.

## 📝 개발 체크리스트 (APFS)

- [ ] **아이콘 전용 조작요소**에 접근名 — `IconBtn label` 또는 `aria-label` (bare 클릭 div 금지)
- [ ] **모든 입력**에 프로그램적 라벨 — `Field`(`<label>`) 래핑 또는 `aria-label`. **placeholder만 두지 말 것**(placeholder는 접근名 아님)
- [ ] **의미 태그** 사용 — `<button>`·`<nav>`·`<h1~6>`·`<ul>`
- [ ] **키보드 단독**으로 전 기능 도달·실행(Tab/Enter/Esc/화살표), 모달 포커스 트랩·닫힘 복귀
- [ ] **초점 가시성** — 전역 `:focus-visible` 링 유지(`outline:none` 금지)
- [ ] `<img>` `alt`(장식은 `alt=""`), 정보 차트는 대체 텍스트/표
- [ ] **동적 상태**를 라이브리전(`Alert`/`Skeleton`/`Spinner`)으로 알림
- [ ] **색+텍스트 병행**, 명도 대비 4.5:1(양 테마), color-tokens 준수
- [ ] **400% 확대/리플로우**에서 가로 스크롤 없이 읽힘(responsive-ui와 함께)
- [ ] 폼 오류를 텍스트로 안내 + 오류 필드 초점 이동
- [ ] Lighthouse(Accessibility) 또는 axe DevTools 통과 — **라이트·다크 모두**
