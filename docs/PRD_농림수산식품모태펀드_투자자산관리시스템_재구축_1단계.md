## 1. 문서 개요

### 1.1 목적

본 문서는 "농림수산식품모태펀드 투자자산관리시스템 재구축(1단계)" 사업의 제품 요구사항을 정의한다. 제안서에 기술된 비전과 기능 구상을 검증 가능하고 추적 가능한 요구사항으로 변환하여, 설계·개발·검수 전 단계에서 공통 기준으로 활용하는 것을 목적으로 한다.

본 문서는 다음을 지향한다.

- 발주처(농업정책보험금융원 등)와 수행사 간 범위·수용 기준에 대한 합의 근거 제공
- 제안서의 정성적 표현을 측정 가능한 수용 기준(Acceptance Criteria)으로 구체화
- 각 요구사항의 우선순위 부여를 통한 단계적 구축 의사결정 지원

### 1.2 범위 (1단계 재구축)

본 문서가 다루는 범위는 제안서 기준 다음 영역의 재구축이다.

| 구분 | 포함 범위 |
|------|-----------|
| 핵심 업무 서비스 (Core Biz) | 회계 관리(자금관리·분개/전표·결산관리), 자펀드보고(운용기관정보·재무/조합정보·자펀드검증), 수탁보고(모태펀드수탁·자펀드수탁) |
| 지원 업무 서비스 (Support) | 투자자산관리(모태/자펀드관리·사후관리·투자기업정보), 조기경보(리스크모니터링·가치평가 관리), 부처보고(등록원부 승인·증명서 발급·투자 현황 통계) |
| 전사 공통 기반 (Foundation) | API, Admin(통합 모니터링·권한 관리·시스템 관리), Login |
| 데이터 | 통합 데이터 플랫폼(Data Lake) 및 6개 시스템 통합 DB, 데이터 마트 구성 |
| 인프라 | 민간(Private) 클라우드 기반 인프라, CSAP 보안 영역 |
| 외부 연계 | NICE 평가정보, 가치평가, 운용사 ERP, 수탁기관 연계 |

> 범위 제외/후속 단계 항목은 제안서에 명시되지 않았으므로, 단계 분리 기준은 발주처와 별도 협의 후 확정한다.

### 1.3 대상 독자

| 독자 | 활용 목적 |
|------|-----------|
| 발주처 사업 담당자 | 요구사항 합의, 검수 기준 확인 |
| 주무관청(농림축산식품부·해양수산부) 업무 담당자 | 부처보고·정책 이행 요건 검토 |

| 아키텍트/설계자 | 시스템 아키텍처 및 데이터 모델 설계 기준 |
| 개발자 | 기능 구현 기준, 수용 기준 충족 확인 |
| QA/검수 담당자 | 단위·통합·인수 테스트 기준 |
| 보안/감리 담당자 | 보안 요구사항 및 컴플라이언스 충족 여부 검증 |

### 1.4 용어 정의

| 약어/용어 | 정의 |
|-----------|------|
| GP (General Partner) | 운용사. 자펀드(투자조합)를 운용·보고하는 주체 |
| LP / 출자자 | 모태펀드 등 출자 주체 (본 시스템에서 모태펀드 측이 해당) |
| RBAC (Role-Based Access Control) | 역할 기반 접근 제어. 본 사업은 슈퍼관리자/운용심사역/일반사용자 3등급 적용 |
| 2-Factor (2-Factor Authentication) | 2단계(이중) 인증 |
| IRR (Internal Rate of Return) | 내부수익률. 투자성과 정량 평가 지표 |
| ROI (Return on Investment) | 투자수익률 |
| AUM (Assets Under Management) | 운용자산 규모 |
| Capital Call | 출자금 요청. 조합이 출자자에게 자금 납입을 요청하는 행위(회계 연동 대상) |
| RTO (Recovery Time Objective) | 목표 복구 시간 (본 사업 목표: 4시간) |
| RPO (Recovery Point Objective) | 목표 복구 시점 (본 사업 목표: 10분) |
| CSAP (Cloud Security Assurance Program) | 클라우드 보안인증. KISA 주관 공공 클라우드 보안 인증 제도 |
| CSP (Cloud Service Provider) | 클라우드 서비스 제공자 (본 사업: Naver 또는 NHN Cloud) |
| eGovFrame | 전자정부 표준 프레임워크 (본 사업 적용 버전: 4.2) |
| DMZ (Demilitarized Zone) | 외부망과 내부망 사이의 완충 네트워크 영역 |
| WAF (Web Application Firewall) | 웹 방화벽 |
| IPS / IDS | 침입방지시스템 / 침입탐지시스템 |
| Data Lake | 통합 데이터 플랫폼. 6개 시스템 데이터를 통합 적재 |
| Cross-Check | 운용사·수탁기관 데이터 간 자동 교차 검증 |
| SLA (Service Level Agreement) | 서비스 수준 협약 (CSP 기술지원 목표: 99.9%) |
| KWCAG | 한국형 웹 콘텐츠 접근성 지침 (본 사업 적용 버전: 2.2) |
| GNB / LNB | Global Navigation Bar / Local Navigation Bar |

## 2. 사업 배경 및 목표

### 2.1 배경

현행 투자자산관리 환경은 운용사·수탁기관·주무관청·농금원 등 다수 이해관계자의 데이터가 분산되어 있고, 클라이언트/서버 컴퓨팅 환경과 분산 데이터베이스 관리에 보안·정합성 취약 요소가 존재한다(제안서 SLIDE 34 근거). 본 사업은 민간(Private) 클라우드 기반의 통합 아키텍처와 통합 데이터 모델을 적용하여, 분산·단절된 데이터를 통합하고 보안성·확장성·가용성을 확보한 투자자산 관리 플랫폼을 재구축하는 것을 배경으로 한다.

### 2.2 목표

제안서의 비전("보안성·확장성·가용성이 보장되는 미래형 투자자산 관리 플랫폼", "데이터 기반의 의사결정 지원")을 다음과 같이 측정 가능한 목표로 정의한다. 마케팅 수사("완벽", "획기적", "100% 보장")는 검증 가능한 기준으로 환산하였다.

| 목표 영역 | 측정 가능한 목표 (수용 기준) | 우선순위 |
|-----------|------------------------------|----------|
| 클라우드 기반 인프라 | 민간(Private) 클라우드 + CSAP 인증 영역에 시스템 구축. CSAP 인증 요건 및 공공기관 보안 가이드라인 항목별 충족 여부를 체크리스트로 입증 | 필수(High) |
| 데이터 통합 | 6개 시스템(투자자산관리·조기경보·수탁보고·자펀드보고·부처보고·회계) 데이터를 통합 DB 및 Data Lake로 통합하고, 데이터 마트를 구성. 시스템 간 동일 데이터 항목의 중복·불일치 0건을 검증 | 필수(High) |
| 가용성 | 장애 복구 목표 RTO 4시간, RPO 10분 충족. DR 리허설 연 1회 수행하여 목표 달성 여부 측정 | 필수(High) |
| 확장성 | 트래픽 급증 시 Auto Scaling 자동 적용, DB 리소스 Scale-up 지원. 오픈 전 부하 테스트(Stress Test)로 목표 응답속도 충족 여부 검증 | 필수(High) |
| 보안성 | 3중 보안 체계(네트워크·시스템·데이터) 적용, DB 암호화·전송구간 TLS 적용, RBAC 3등급 + 2-Factor 인증 적용. 작업자 실명·IP·변경 전후 데이터의 감사 로그 기록 누락 0건 | 필수(High) |
| 데이터 기반 의사결정 | KPI 중심 대시보드 제공, IRR/ROI/AUM 등 정량 지표 시각화. 주요 업무 화면에서 핵심 지표를 단일 화면(One-View)에서 조회 가능 | 권장(Medium) |
| 데이터 무결성 | 운용사·수탁기관 데이터 자동 교차 검증(Cross-Check) 적용. 장부 금액과 실물 잔고 불일치 시 담당자 자동 알림 발생 | 필수(High) |
| 표준 준수 | eGovFrame 4.2 호환성 가이드 준수, 정보시스템 구축·운영 지침, 농업정책보험금융원 표준 가이드 및 공공기관 DB 표준화 지침을 항목별로 준수 입증 | 필수(High) |

> 비고: 제안서의 "100% 보장/완벽/획기적/무결점" 등 표현은 본 PRD에서 측정 기준(항목별 충족 입증, 불일치 0건, 목표치 달성 검증)으로 대체하였다.

## 3. 이해관계자 및 사용자

### 3.1 외부 이해관계자

| 이해관계자 | 역할/책임 | 시스템 접점 |
|------------|-----------|-------------|
| 운용사 / GP | 자펀드(조합) 운용, 정기·수시 보고 작성, 투심 결과·총회 결과 등록, Capital Call 요청, 등록원부 작성, 소명 자료 제출 | 운용사 보고 입력 화면, ERP 연계, 등록원부 작성 |
| 수탁기관 | 수탁 계좌·예치금·입출금 데이터 제공, 실물자료 업로드, 유가증권 정보 제공 | 수탁보고 업로드, Cross-Check 데이터 연계 |
| 농림축산식품부 · 해양수산부 (주무관청) | 등록원부 최종 결재·승인, 정책 이행 검증, 부처보고 수신 | 부처보고 승인 화면, 등록원부 Locking 승인 |
| 농업정책보험금융원 (농금원) | 모태펀드 운용·관리, 월간 보고 양식 표준 제공, 데이터 표준 가이드 적용 주체 | 출자/분배 조회(농금원), 월간 보고 양식 매핑 |
| NICE 평가정보 | 투자기업 신용정보·신용등급 원천 제공 | API 연계(조기경보 신용정보) |
| 가치평가 기관 | 가치평가 결과 데이터 제공 | API/연계(가치평가 결과 관리) |

### 3.2 시스템 사용자 (RBAC 3등급)

제안서 SLIDE 19 기준, 본 시스템은 RBAC 3등급 권한 체계를 적용한다.

| 권한 등급 | 역할/책임 | 시스템 접점(접근 범위) |
|-----------|-----------|------------------------|
| 슈퍼관리자 | 전체 접근. 공통코드·메뉴·도움말·사용자·권한 관리, 외부연동 관리, 보안(개인정보 접속관리·로그 조회), 동적 운영 환경 설정 | 관리자시스템 전체, 전사 공통 기반(Admin) |
| 운용심사역 | 심사/평가 업무. 투심 보고 검토·승인, 사후관리 검증, 조기경보 처리, 운용사 평가, 부처보고 검토 | 핵심·지원 업무 서비스 중 심사/평가 권한 메뉴 |
| 일반사용자 | 조회 전용. 대시보드·통계·명세서 등 조회 | 조회 권한 부여 메뉴(읽기 전용) |

수용 기준:
- 운용사별 접근 제어가 적용되어 자사 정보 외 타 운용사 데이터 접근이 차단됨(SLIDE 13 근거).
- 권한 등급별 메뉴/기능 노출이 동적으로 제어되고, 권한 외 기능 접근 시도가 차단·기록됨.
- 모든 사용자 인증에 2-Factor 인증이 적용됨.

## 4. 시스템 아키텍처 개요

본 장은 제안서 SLIDE 2(목표시스템 구성도)를 텍스트로 구조화한 것이다. 구체 인프라 사양(서버 사이징, CSP, RTO/RPO 등)은 6장 비기능 요구사항에서 상세화하며, 본 장은 논리 구조와 영역 구분을 정의한다.

### 4.1 네트워크 영역 구분

| 영역 | 구성 요소 | 설명 |
|------|-----------|------|
| DMZ | External Gateway | 외부망과의 접점. 외부 연계 트래픽 및 외부 사용자 접근 처리 |
| Secure Private Cloud Zone (CSAP) | Internal Gateway, 전사 공통 기반, 데이터 플랫폼, 업무 서비스 | CSAP 인증 영역. 핵심 업무·데이터·인증이 위치하는 보안 영역 |

External Gateway와 Internal Gateway의 2단 게이트웨이 구조로 외부망과 내부 보안 영역을 분리한다.

### 4.2 전사 공통 기반 (Foundation)

통합 모니터링, 권한 관리, 시스템 관리를 담당하는 공통 계층으로 다음 컴포넌트로 구성된다.

- **API**: 내부 서비스 및 외부 연계의 API 처리
- **Admin**: 통합 모니터링, 권한 관리(RBAC), 시스템 관리(공통코드·메뉴·도움말·사용자 관리)
- **Login**: 인증/로그인 처리 (2-Factor 인증 연동)

### 4.3 통합 데이터 플랫폼 (Data Lake)

6개 업무 시스템의 데이터를 통합 적재하는 데이터 플랫폼이다. 데이터 마트 구성을 통해 통계·실적 정보를 통합 제공한다(SLIDE 29 근거). 핵심·지원 업무 서비스가 공통으로 참조하는 통합 데이터 기반 역할을 한다.

## 5. 기능 요구사항

본 장은 투자자산 관리 핵심 업무 영역의 기능 요구사항을 정의한다. 각 요구사항은 검증 가능한 수용 기준(AC)을 동반하며, 제안서의 강조도에 따라 우선순위를 부여한다.

### 5.1 모태펀드 관리 (FR-5.1)

**개요**: 출자사업의 공고-심사-선정-결성에 이르는 전주기(Lifecycle)를 단일 흐름으로 연결하여 데이터 단절 없이 관리하고, 재원 조성·출자 집행을 회계 시스템과 연동한다. 대용량 데이터의 일괄 처리와 업무 이력 추적을 지원한다.

**주요 사용자**: 모태펀드 운용 담당자(농금원), 투자심의위원회, 시스템 관리자

**세부 요구사항**

| ID | 요구사항 |
|----|----------|
| FR-5.1-01 | 출자사업 분야별 공고를 등록·수정·조회할 수 있으며, 모든 변경에 대해 변경 이력을 자동 기록한다. |
| FR-5.1-02 | 서류심사 평가기준을 항목별로 설정·저장하고, 심사보고서를 작성·관리한다. |
| FR-5.1-03 | 투자심의위원회 정보를 등록·관리하고, 심의 선정 결과를 후속 결성 프로세스로 연동한다. |
| FR-5.1-04 | 선정 운용사의 결성 승인 처리와 결성 기한 연장을 수행하며, 승인·연장 이력을 추적한다. |
| FR-5.1-05 | 모태펀드 투심위 개최 및 조합 결성 관리를 위한 의사결정 지원 화면을 제공한다. |
| FR-5.1-06 | 기금별 출자·배분 계획을 수립하고 데이터를 입력·수정할 수 있다. |
| FR-5.1-07 | 모태펀드 재원 조성 및 출자 집행 현황을 회계 시스템과 연계하여 처리하고, 실시간 대시보드로 조회한다. |
| FR-5.1-08 | 대용량 엑셀 파일을 일괄(Batch) 업로드하여 등록하고, 표준 입력 양식을 다운로드할 수 있다. |
| FR-5.1-09 | 각 레코드 단위로 메모 및 비고를 입력·통합 관리하고, 업무 이력을 통합 조회한다. |

**수용 기준(AC)**
- 공고/심사/선정/결성 단계 전환 시, 직전 단계에서 입력된 데이터가 재입력 없이 다음 단계로 승계된다.
- 결성 승인 또는 기한 연장 1건마다 작업자·일시·변경 전후 값이 이력 레코드로 생성되어 조회된다.
- 엑셀 일괄 업로드 시 양식 불일치/검증 실패 행은 처리 거부되고, 실패 행 번호와 사유가 결과로 반환된다(부분 성공 허용 정책은 설계 시 확정).
- 다운로드한 양식으로 작성·업로드한 파일은 별도 변환 없이 정상 적재된다.
- 출자 집행 처리 결과가 회계 시스템 연동 데이터와 일치한다(불일치 시 처리 보류).

**우선순위**: High (제안서 1.2.1 핵심 기능, 전주기 원스톱 강조)

### 5.2 사후 보고 관리 (FR-5.2)

**개요**: 운용사(GP)와 심사역 간 양방향 보고 채널을 구축하여 투자심의(Step 1), 운용보고(Step 2), 의사결정(Step 3)의 사후관리 전 과정을 처리하고, 출자금 요청(Capital Call)을 운용사 ERP 및 모태펀드 회계와 연동한다.

**주요 사용자**: 운용사(GP), 담당 심사역, 팀장(승인권자), 준법감시 담당자

**세부 요구사항**

| ID | 요구사항 |
|----|----------|
| FR-5.2-01 | 운용사는 투심 일정·결과를 등록하고 의사록 파일을 첨부할 수 있다. |
| FR-5.2-02 | 투자심의 승인은 "담당자 검토 → 팀장 승인 → 확정"의 다단계 워크플로우로 처리한다. |
| FR-5.2-03 | 준법감시 항목(규약 준수 여부)에 대한 필수 체크리스트 검증을 거치지 않으면 다음 단계로 진행할 수 없다. |
| FR-5.2-04 | 투심 위원 구성에 따른 정족수 및 의결권을 자동 계산하여 표시한다. |
| FR-5.2-05 | 운용보고는 정기 보고와 수시 보고를 구분하여 작성·제출할 수 있다. |
| FR-5.2-06 | 전월 대비 데이터 변동을 자동 비교하여 변동 항목을 화면상 하이라이팅한다. |
| FR-5.2-07 | 보고서 수정 이력 및 버전을 관리하여 이전 버전을 추적·조회할 수 있다. |
| FR-5.2-08 | 조합원 총회 안건·결과를 등록하고, 총회 일정·결과를 확정 처리한다. |
| FR-5.2-09 | 담당 심사역은 검토 의견서를 작성하고 승인 처리할 수 있다. |
| FR-5.2-10 | 총회 일정·결과의 상태를 확정/미확정/제외로 구분하여 관리한다. |
| FR-5.2-11 | 조합 예상 자금 요청(Capital Call) 내역을 운용사 ERP로 자동 전송하고, 모태펀드 회계 집행과 연동 관리한다. |

**수용 기준(AC)**
- 워크플로우 단계는 정의된 순서로만 진행되며, 선행 단계 미완료 시 후행 단계 진입이 거부된다.
- 준법감시 필수 체크리스트 항목 중 하나라도 미충족이면 "확정" 처리가 거부되고 미충족 항목이 표시된다.
- 정족수 미달 시 의결 확정이 불가하며, 시스템이 계산한 정족수·의결권 값이 입력 위원 구성과 일치한다.
- 전월 대비 값이 변경된 항목은 변동 표시(하이라이팅) 대상으로 식별되며, 변동 없는 항목은 표시되지 않는다.
- 보고서 수정 시 새 버전이 생성되고 직전 버전이 보존되어 버전 간 차이를 조회할 수 있다.
- Capital Call 전송 시 운용사 ERP로의 전송 결과(성공/실패)가 기록되며, 전송된 금액이 회계 집행 요청 금액과 일치한다.

**우선순위**: High (제안서 1.2.2 핵심, 워크플로우·검증·연동 다수 강조)

### 5.3 자펀드 정보 관리 (FR-5.3)

**개요**: 수탁기관 데이터를 기준으로 자펀드 장부금액과 실물잔고를 자동 교차검증(Auto Cross-Check)하여 데이터 무결성을 확보한다. 일치 시 자동 승인, 불일치 시 담당자 알림으로 처리하며, 은행 거래원장 업로드와 다차원 실적분석·보고서 출력을 지원한다.

**주요 사용자**: 자펀드 관리 담당자(심사역), 수탁기관, 회계 담당자

**세부 요구사항**

| ID | 요구사항 |
|----|----------|
| FR-5.3-01 | 자펀드 원장 정보(회계 원장 자동 생성, 출자·분배 데이터, 운용 지시 내역, 자금 집행 요청 및 변경 이력)를 관리한다. |
| FR-5.3-02 | 수탁은행 계좌의 실제 예치금 잔액 및 계좌 상태를 조회한다. |
| FR-5.3-03 | 은행 거래원장(입금/출금)을 엑셀로 업로드하여 실제 입출금 내역을 적재한다. |
| FR-5.3-04 | 장부금액과 실물잔고를 교차검증하여, 일치(CASE A) 시 해당 건을 자동 승인 처리한다. |
| FR-5.3-05 | 금액 차이 또는 날짜 불일치(CASE B) 발생 시 해당 건을 담당자에게 즉시 알림으로 통지하고 검토 대상으로 분류한다. |
| FR-5.3-06 | 산업별·분야별·연도별 다차원 실적 통계를 시각화하여 제공한다. |
| FR-5.3-07 | 투자실적보고서를 원클릭으로 생성·출력할 수 있다. |

**수용 기준(AC)**
- 교차검증 시 "장부금액 == 실물잔고" 조건이 성립하면 해당 건은 자동 승인 상태로 전환된다.
- 금액 차이 또는 거래일자 불일치가 1건이라도 존재하면 자동 승인되지 않고, 담당자에게 알림이 발송되며 불일치 항목(차이 금액/일자)이 기록된다.
- 엑셀로 업로드한 입출금 내역이 검증 대상 데이터로 적재되어 교차검증에 반영된다.
- 실적 통계는 산업/분야/연도 조건 조합으로 필터링되며, 집계 값이 원천 데이터 합계와 일치한다.
- 투자실적보고서 출력 시 화면에 표시된 실적 데이터와 보고서 내용이 일치한다.

> 비고: 제안서의 "데이터 무결성을 100% 보장" 표현은 측정 불가 수사이므로, 본 요구사항에서는 "교차검증을 통과하지 못한 건은 자동 승인되지 않고 담당자 검토로 전환된다"는 검증 가능한 통제 기준으로 대체한다.

**우선순위**: High (제안서 1.2.3 핵심, 자동 교차검증 강조)

### 5.4 투자기업 정보 조회 (FR-5.4)

**개요**: 투자기업의 재무·고용·회수 성과를 통합 분석하는 360° 기업 포트폴리오를 제공한다. 기업 명세서, 예비 유니콘 선정, 고용창출 효과 분석, 우수 사례(Best Practice) 및 노하우 축적(Knowledge Base)을 지원한다.

**주요 사용자**: 심사역, 모태펀드 운용 담당자, 정책 담당 부처/관리자

**세부 요구사항**

| ID | 요구사항 |
|----|----------|
| FR-5.4-01 | 투자기업별 기본정보(설립일, 대표이사, 주요주주 지분율 등)와 재무 지표(매출액, 전년 대비 증감률, 재무 건전성 등급)를 통합한 기업 명세서를 제공한다. |
| FR-5.4-02 | 고용창출 효과를 투자 전 대비 고용 비율(예: 투자 전 대비 145%)로 산출·표시한다. |
| FR-5.4-03 | 연도별 회수 현황을 ROI 및 목표 수익률(IRR) 기준으로 분석·표시한다. |
| FR-5.4-04 | 정의된 기준에 따라 예비 유니콘 선정 기업을 식별·관리한다. |
| FR-5.4-05 | 우수 사례(Monthly Best Practice)를 선정·등록하고, 성공 요인·노하우 리포트를 Knowledge Base에 축적·조회한다. |

**수용 기준(AC)**
- 기업 명세서의 재무·고용·회수 지표는 원천 데이터로부터 산출되며, 표시 값이 원천 값과 일치한다.
- 고용창출 효과는 "(현재 고용 ÷ 투자 전 고용) × 100(%)" 산식으로 계산되며(예: 145%), 산식과 표시 값이 일치한다. 표기 방식(비율 vs 증가율)은 업무 규칙 확정 후 최종 반영한다.
- IRR/ROI 값은 정의된 계산식에 따라 산출되며, 동일 입력에 대해 재계산 시 동일 결과를 반환한다.
- 예비 유니콘 선정 기준에 부합하는 기업만 선정 목록에 포함된다.
- Best Practice 리포트 등록 시 Knowledge Base에서 검색·조회 가능하다.

> 비고: 제안서의 "유니콘 기업 육성", "성장 지원 실현" 등은 사업 비전 문구이므로, 본 요구사항에서는 식별·분석·축적 기능으로 한정하여 정의한다.

**우선순위**: Medium (제안서 1.2.4, 조회·분석 중심으로 핵심 트랜잭션 대비 상대적 낮음)

### 5.5 운용사 보고 정보 조회 (FR-5.5)

**개요**: 운용 성과에 기반한 정량 평가와 운용사 보고 내역의 상호 검증(Cross-Check)을 통해 운용사 건전성을 모니터링하고, 규정에 따른 관리보수를 산정·확정한다.

**주요 사용자**: 농금원/주무관청 심사역, 운용사 관리 담당자, (입력 측) 운용사(GP)

**세부 요구사항**

| ID | 요구사항 |
|----|----------|
| FR-5.5-01 | 운용사 건전성 Dashboard에 운용사별 핵심 지표(설립일, AUM, 신용등급, 성과평가 등급)를 실시간 조회 형태로 표시한다. |
| FR-5.5-02 | 운용 활동 검증 체크리스트를 제공하여 분기별 실사보고서 제출일, 관리보수 지급 기준 충족 여부, 규정 준수 확인 상태(Pass/Fail)를 항목별로 관리한다. |
| FR-5.5-03 | 투자금 의무 집행 비율을 목표치(Target) 대비 달성률로 산출·표시한다. (예: Target 80% 대비 달성 80%) |
| FR-5.5-04 | 보수 정산 계산기를 제공하여 `기본 관리보수(Base) + 성과 인센티브(성과등급 연동) − 운용 규정 위반 삭감 = 최종 지급 확정액`을 자동 계산한다. |
| FR-5.5-05 | 보수 정산 결과는 운용사 보고 내역(실사보고서, 성과평가 등급, 위반 이력)과 연계하여 산정 근거를 항목별로 추적할 수 있어야 한다. |
| FR-5.5-06 | 규정 준수 확인 후 담당자가 "확정 승인" 처리를 할 수 있으며, 승인 시 해당 정산 건의 상태가 확정으로 전환된다. |

**수용 기준(AC)**
- AC-5.5-1: Dashboard 지표(AUM·신용등급·성과평가)는 원천 데이터 변경 후 재조회 시 최신 값으로 반영된다.
- AC-5.5-2: 보수 정산 계산식에 기본 관리보수 50,000,000, 성과 인센티브 20,000,000, 규정 위반 삭감 5,000,000을 입력하면 최종 지급 확정액 65,000,000원이 산출된다.
- AC-5.5-3: 의무 집행 비율이 목표치 미달일 경우 미달 상태가 식별되어 화면에 구분 표시된다.
- AC-5.5-4: "확정 승인" 전에는 정산액이 미확정 상태로 표시되며, 승인 후에는 산정 근거(구성 항목)가 함께 조회된다.

**우선순위**: High (성과관리 핵심 기능, 보수 지급 확정과 직결)

### 5.6 조기경보 관리 (FR-5.6)

**개요**: 다차원 리스크 지표와 외부 신용정보(NICE 평가정보 API)를 연계하여 부실 징후를 포착하고, 위반 감지부터 마감까지 단계별 리스크 소명·해소 절차(Risk Resolution Process)를 관리한다.

**주요 사용자**: 리스크 관리 담당자(농금원/주무관청), 소명 자료 제출 측 운용사(GP)

**세부 요구사항**

| ID | 요구사항 |
|----|----------|
| FR-5.6-01 | NICE 평가정보 API를 연동하여 신용등급 변동, 채무불이행, 법정관리 신청, 현금흐름 등급 등 외부 신용정보를 원천 데이터로 수집한다. |
| FR-5.6-02 | 수집된 외부 신용정보와 내부 규약 준수 데이터를 결합하여 리스크 판정을 수행한다. |
| FR-5.6-03 | Risk Resolution Process 5단계를 시스템화한다: (1) 위반 감지(Detect, 자동) (2) 리스크 등록(Register) (3) 소명 및 해소(Action) (4) 해제 등록(Release) (5) 마감(Close). |
| FR-5.6-04 | 1단계: 규약 위반·신용 하락 등 부실 징후를 시스템이 자동 포착하고 담당자에게 알림을 발송한다. |
| FR-5.6-05 | 2단계: 감지된 위반 사항의 상세 내용 및 증빙 자료를 시스템에 등록한다. |
| FR-5.6-06 | 3단계: 운용사 소명 자료를 접수하고 해소 조치(자금 보전, 변경 계약 등) 이행 내역을 기록한다. |
| FR-5.6-07 | 4단계: 리스크 요인 해소 확인 후 정상 상태로 전환하는 해제 승인 및 이력을 관리한다. |
| FR-5.6-08 | 5단계: 해당 건을 최종 종결 처리하고 사후 평가 데이터에 반영한다. |
| FR-5.6-09 | 각 단계 전환 시 위반 사항 추적 이력을 보존한다. |

**수용 기준(AC)**
- AC-5.6-1: NICE 평가정보 API에서 신용등급 변동 또는 법정관리 신청 등 트리거 이벤트가 수신되면 해당 운용사/자펀드에 대한 리스크 후보 건이 생성된다.
- AC-5.6-2: 1단계 위반 감지 시 담당자에게 알림이 발송된 기록이 시스템에 남는다.
- AC-5.6-3: 5단계 흐름은 순차적으로만 진행되며, 등록(2단계) 없이 해제(4단계)로 전환할 수 없다.
- AC-5.6-4: 4단계 해제 등록 시 해소 확인 근거가 입력되지 않으면 정상 상태 전환이 거부된다.
- AC-5.6-5: 마감(5단계) 처리된 건은 사후 평가 데이터 조회 시 포함된다.

**우선순위**: High (선제적 부실 대응, 외부 신용 연계 핵심)

### 5.7 가치평가 결과 관리 (FR-5.7)

**개요**: 실시간 조기경보 대시보드를 통해 운용사 상태와 리스크 지수 추이를 시각화하고, IRR(내부수익률) 입체 분석 및 자펀드 대비 벤치마크 수익률 비교를 제공한다.

**주요 사용자**: 리스크 관리 담당자, 심사역, 성과 분석 담당자

**세부 요구사항**

| ID | 요구사항 |
|----|----------|
| FR-5.7-01 | Risk Dashboard에 운용사별 상태를 정상/주의/경고 3단계로 구분 표시한다. |
| FR-5.7-02 | 리스크 지수 추이를 시계열(Time Series) 그래프로 제공한다. |
| FR-5.7-03 | 수익률(IRR) 입체 분석을 투자기업별·자펀드별·계약별 관점으로 제공한다. |
| FR-5.7-04 | 투자기업별 IRR 상위 항목(예: Top 5)을 순위 형태로 표시한다. |
| FR-5.7-05 | 자펀드 수익률과 벤치마크(BM) 수익률을 동일 기준으로 비교 표시한다. |
| FR-5.7-06 | 주요 계약별 평가 상세(투자기업명, 계약일자, 투자금액, 평가금액, IRR)를 목록으로 조회한다. |

**수용 기준(AC)**
- AC-5.7-1: 운용사 상태 값은 리스크 판정 결과에 따라 정상/주의/경고 중 하나로 일관되게 표시된다.
- AC-5.7-2: IRR 분석 관점을 투자기업별/자펀드별/계약별로 전환하면 해당 기준으로 집계된 결과가 표시된다.
- AC-5.7-3: 계약별 평가 상세에서 투자금액·평가금액·IRR이 동일 계약 데이터로 정합되게 표시된다. (예: 계약일 2023.05.10, 투자금액 30억원, 평가금액 45억원, IRR +32.5%)
- AC-5.7-4: 자펀드 vs 벤치마크 비교는 동일 기간 기준의 두 수익률을 함께 표시한다.

**우선순위**: High (성과·리스크 시각화 핵심)

### 5.8 등록원부 관리 (FR-5.8)

**개요**: 부처 보고·대관 업무를 위한 등록원부를 관리하며, 부처 승인 완료 시 시스템 레벨에서 수정을 차단하는 승인 통제(Locking)로 데이터 무결성을 확보한다.

**주요 사용자**: 운용사(작성), 시스템 접수 담당자, 주무관청(농식품부·해수부) 승인권자

**세부 요구사항**

| ID | 요구사항 |
|----|----------|
| FR-5.8-01 | 등록원부 처리 흐름을 4단계로 관리한다: (1) 운용사 작성 (2) 시스템 접수 (3) 주무관청 승인 (4) Lock/수정불가. |
| FR-5.8-02 | 1단계: 운용사가 등록원부를 입력·수정 신청하면 이력이 자동 생성된다. |
| FR-5.8-03 | 2단계: 시스템 접수 시 데이터 형식 검증 및 작성 권한을 확인한 후 접수 처리한다. |
| FR-5.8-04 | 3단계: 주무관청(농식품부·해수부)이 최종 결재·승인하는 승인 통제 절차를 제공한다. |
| FR-5.8-05 | 승인 후 수정 불가(Locking): 부처 승인 완료 시 시스템 레벨에서 수정 기능을 비활성화하고 해당 데이터를 Read-only로 전환·동결한다. |
| FR-5.8-06 | 운용사별 접근 제어: 자사 정보 외 타 운용사 데이터 접근을 차단한다. |
| FR-5.8-07 | 이력 추적: 최초 등록부터 최종 승인까지 변경 이력(History)을 영구 보관한다. |
| FR-5.8-08 | 대량 등록: 엑셀(Excel) 업로드를 통한 일괄 등록(Batch) 처리를 지원한다. |
| FR-5.8-09 | 공식 문서 발급: 승인된 등록원부를 PDF로 변환하여 다운로드·증빙 활용할 수 있도록 한다. |

**수용 기준(AC)**
- AC-5.8-1: 부처 승인이 완료된 등록원부에 대해 수정 API를 호출하면 거부되며, 화면에서도 Read-only로 표시된다.
- AC-5.8-2: 운용사 계정으로 타 운용사의 등록원부 데이터를 조회·접근 시도하면 접근이 차단된다.
- AC-5.8-3: 등록원부의 최초 등록·수정·승인 행위가 각각 이력으로 기록되며, 이력은 삭제되지 않는다.
- AC-5.8-4: 형식 검증을 통과하지 못한 행이 포함된 엑셀 업로드는 해당 오류 건을 식별하여 접수되지 않는다.
- AC-5.8-5: 승인 완료 상태의 등록원부만 PDF 공식 문서로 발급된다.

**우선순위**: High (대관 업무·데이터 무결성 핵심, 승인 통제 필수)

### 5.9 통계조회 (FR-5.9)

**개요**: 부처 보고를 위한 통계를 통합 조회하며, 투자기업 현황 보고(Reporting), 의무 투자 비율 준수 모니터링(Compliance), 다차원 포트폴리오 분석(Analytics)을 제공한다.

**주요 사용자**: 대관 업무 담당자, 부처 보고 담당자, 정책·성과 분석 담당자

**세부 요구사항**

| ID | 요구사항 |
|----|----------|
| FR-5.9-01 | Reporting: 신규/전체 투자기업 목록과 투자 단계별 기업 리스트 및 상세 정보를 조회한다. |
| FR-5.9-02 | Reporting: 농금원 '월간 보고' 표준 양식에 맞춰 데이터를 자동 추출·변환(자동 매핑)한다. |
| FR-5.9-03 | Compliance: 연도별 의무 투자 비율 준수 현황을 펀드 규약 기준 목표 대비 달성률로 조회한다. |
| FR-5.9-04 | Compliance: 주목적 투자 비율을 모니터링하고, 위반 가능성 발생 시 조기 경보(Alert) 및 알림을 발송한다. |
| FR-5.9-05 | Analytics: 지역(시/군) 및 산업별 투자 비중을 분석·시각화한다. |
| FR-5.9-06 | Analytics: 기간·펀드·산업·지역 등 다양한 조건을 조합하는 다차원 필터링을 제공한다. |

**수용 기준(AC)**
- AC-5.9-1: 농금원 월간 보고 양식 매핑 결과는 원천 데이터와 항목 단위로 일치한다.
- AC-5.9-2: 연도별 의무 투자 비율 조회 시 규약 기준 목표치와 실제 달성률이 함께 표시된다.
- AC-5.9-3: 주목적 투자 비율이 위반 임계치에 도달하면 경보(Alert)가 발생하고 알림 발송 기록이 남는다.
- AC-5.9-4: 다차원 필터(기간·펀드·산업·지역)를 조합하면 조건에 부합하는 결과만 집계되어 표시된다.

**우선순위**: Medium (조회·모니터링 중심. 단, Compliance 경보 항목은 High)

### 5.10 회계 자금관리 (FR-5.10)

**개요**: (SLIDE 15~16 근거) 복식부기 기반의 자동 분개 엔진과 캘린더 뷰 기반 마감 관리를 통해 모태펀드 회계 데이터의 무결성을 확보하고, 자금 원천별 회계 독립성과 승인 기반 통제, 실시간 자금수지(Cash Flow) 모니터링을 제공한다.

**주요 사용자**: 회계 담당자, 회계 승인권자(팀장 등), 자금관리 담당자

**세부 요구사항**

| ID | 요구사항 |
|----|----------|
| FR-5.10-01 | 거래유형별 표준 분개 룰(Rule)을 정의하고, 입출금 발생 시 해당 룰에 따라 전표를 자동 생성한다(자동 분개 엔진, Auto Journaling). 수기 입력이 필요한 항목은 별도 식별한다. |
| FR-5.10-02 | 자금 원천을 농식품 모태·수산 모태·운영비·기타 사업의 4개로 구분하고, 원천별로 계좌·고정자산·거래처를 별도 관리하여 펀드 간 자금 혼용을 차단한다. |
| FR-5.10-03 | 자금 원천별로 각각 독립된 재무상태표(BS)와 손익계산서(PL)를 생성·조회할 수 있다. |
| FR-5.10-04 | 원천/발생 구분별 전표 승인(미승인) 절차를 시스템화하고, 승인된 전표만 재무제표에 반영한다. 미승인 전표는 재무제표 산출에서 제외한다. |
| FR-5.10-05 | 캘린더 뷰(월간/주간/목록) 기반으로 일자별 마감 상태(일마감 진행중/완료, 휴일)를 시각적으로 표시한다. |
| FR-5.10-06 | 일자별 일마감을 수행하고, 마감 완료 상태를 캘린더에 반영한다(완료 표식 표시). |
| FR-5.10-07 | 자금 원천별 자금수지(Cash Flow) 현황을 실시간으로 조회·모니터링한다. |

**수용 기준(AC)**
- 사전 정의된 거래유형의 입출금 등록 시, 해당 표준 분개 룰에 따른 전표가 자동 생성되어 차변/대변 합계가 일치한다.
- 한 자금 원천의 계좌·거래처는 다른 원천의 전표 입력 선택지에 노출되지 않는다(혼용 차단).
- 4개 자금 원천 각각에 대해 BS와 PL이 독립적으로 산출되며, 원천 간 금액이 합산·교차되지 않는다.
- 미승인 상태의 전표는 재무상태표·손익계산서 등 재무제표 집계에 포함되지 않으며, 승인 처리 후에만 반영된다.
- 캘린더 뷰에서 특정 일자의 마감 상태(미마감/진행중/완료/휴일)가 정확히 구분 표시된다.
- 일마감 완료 처리된 일자는 캘린더에서 완료 표식으로 전환된다.

**우선순위**: High (핵심 구현 기능으로 명시, 자금 무결성 직결)

### 5.11 회계 이력 관리 (FR-5.11)

**개요**: (SLIDE 17 근거) 회계 데이터의 변경 전후 스냅샷(Data Snapshot)을 자동 저장하고, 작업자 신원(WHO)·작업 시점(WHEN)·변경 내용(WHAT)을 기록하여 데이터 조작 리스크를 차단하고 회계 투명성을 보장한다.

**주요 사용자**: 회계 담당자, 회계 승인권자, 감사자(Auditor), 시스템 관리자

**세부 요구사항**

| ID | 요구사항 |
|----|----------|
| FR-5.11-01 | 회계 데이터 변경 시 변경 전(Before)·변경 후(After) 데이터의 스냅샷을 자동 저장한다. |
| FR-5.11-02 | 각 이력에 작업자 신원(WHO: 실명, 사번, IP, 부서)을 기록한다. (예: 김철수 책임 / 부서: 투자관리팀 / 사번: 20230058 / IP: 192.168.10.25) |
| FR-5.11-03 | 각 이력에 작업 시점(WHEN)을 초 단위 타임스탬프로 기록한다. (예: 2026-01-21 14:32:05) |
| FR-5.11-04 | 변경 내용(WHAT)을 변경 항목별 Before → After 형태로 기록한다. (예: 결성총액 100억 → 120억, 존속기간 5년 → 7년) |
| FR-5.11-05 | 작업 유형을 조회(READ)·수정(UPDATE)·승인(APPROVE)으로 구분하여 기록한다. |
| FR-5.11-06 | 기간 및 작업자 기준으로 이력을 필터링 조회하고, 개별 이력의 상세보기를 제공한다. |
| FR-5.11-07 | 조회된 이력 목록을 엑셀 파일로 다운로드한다. |

**수용 기준(AC)**
- 회계 데이터 1건을 수정하면, 변경 전/후 값이 항목 단위로 보존된 이력 1건이 생성된다.
- 생성된 이력에는 실명·사번·IP·부서, 초 단위 타임스탬프, 작업 유형(READ/UPDATE/APPROVE)이 모두 포함된다.
- 조회(READ) 행위도 작업 유형으로 기록되어 이력에 남는다.
- 기간/작업자 필터 적용 시 조건에 맞는 이력만 조회되며, 엑셀 다운로드 결과가 화면 조회 결과와 일치한다.

> 비고: 제안서의 "100% 기록" 표현은 측정 가능한 형태로 번역하여, "데이터 변경 1건당 변경 전/후 스냅샷·작업자 신원·타임스탬프를 포함한 이력 레코드가 누락 없이 생성된다"는 기준으로 정의한다.

**우선순위**: High (감사·내부통제 직결)

### 5.12 농식품경영체 정보 조회 (FR-5.12)

**개요**: (SLIDE 18 근거) 농식품 경영체의 기본정보·재무현황·정부 지원 이력·비정형 IR 자료를 한 화면에 통합(One-View)하여, 정보 탐색 시간을 단축하고 데이터 기반의 투자 검토를 지원한다.

**주요 사용자**: 운용심사역, 투자관리 담당자, 부처/관리기관 담당자

**세부 요구사항**

| ID | 요구사항 |
|----|----------|
| FR-5.12-01 | 경영체 프로필을 통합 화면으로 제공한다(기본정보 + 재무현황 대시보드). 기본정보 항목: 기업명, 영문명, 대표자, 설립일, 기업단계(예: 시리즈 A), 업종 태그(예: 스마트팜, 친환경비료). |
| FR-5.12-02 | 연도별 매출 추이 등 재무현황을 대시보드 형태로 시각화한다(실적 연도 + 추정연도(E) 구분 표기). |
| FR-5.12-03 | IR 자료실(디지털 아카이빙)을 제공하여 비정형 사업계획서·IR Deck 등을 파일로 등록·보관·조회한다. 각 파일에 등록일시·용량·등록자 메타데이터를 기록하고, 파일 목록 필터를 제공한다. |
| FR-5.12-04 | 정부 지원사업 이력을 매핑하여 표시하고, 동일 항목에 대한 중복 수혜 여부를 체크한다(예: 2022 예비창업/창업진흥원, 2024 R&D 지원/농식품부, 사업화 자금 미수혜). |
| FR-5.12-05 | 경영체의 성장 단계를 분석하여 표시한다. |

**수용 기준(AC)**
- 경영체 1건 선택 시 기본정보·재무현황·IR 자료·정부 지원 이력이 단일 화면(One-View)에서 함께 조회된다.
- IR 자료실에 업로드된 파일은 등록일시·용량·등록자가 함께 표시되고, 필터 조건으로 목록이 좁혀진다.
- 정부 지원사업 이력에서 동일 지원 항목의 중복 수혜 발생 시 수혜/미수혜 상태가 구분 표시된다.
- 매출 추이 그래프에서 실적연도와 추정연도(E)가 구분되어 표기된다.

**우선순위**: Medium

### 5.13 관리자 시스템 (FR-5.13)

**개요**: (SLIDE 19 근거) 동적(Dynamic) UI 메뉴 구성을 통한 운영 유연성과 RBAC 기반 권한 통제를 제공하고, 실시간 감사 로그와 접속 통계 분석으로 보안 감사 체계를 구축한다.

**주요 사용자**: 슈퍼관리자, 운용심사역, 일반사용자, 감사자

**세부 요구사항**

| ID | 요구사항 |
|----|----------|
| FR-5.13-01 | 동적 UI 메뉴 구조도(Menu Tree)를 관리하여, 메뉴 변경이 즉시 반영되도록 한다(신규 메뉴 NEW 표식 포함). |
| FR-5.13-02 | 팝업 공지를 활성화/비활성화하고, 시스템 점검 모드로 전환하는 기능을 제공한다. |
| FR-5.13-03 | RBAC 권한 체계를 제공하여 역할을 슈퍼관리자(전체 접근)·운용심사역(심사/평가)·일반사용자(조회 전용)로 구분하고 권한을 차등 부여한다. |
| FR-5.13-04 | 실시간 감사 로그를 기록한다. 대상 행위: 접속(로그인), 개인정보 조회(예: 주민번호 조회), 권한(등급) 변경. 각 로그에 행위 유형·계정 ID·시각을 기록한다. |
| FR-5.13-05 | 접속 통계를 분석하여 일별 사용자 추이를 제공한다. |
| FR-5.13-06 | 접속 기기를 PC/Mobile/Tablet으로 구분하여 비율 통계를 제공한다(예: PC 65% / Mobile 20% / Tablet 15%). |

**수용 기준(AC)**
- 관리자가 메뉴 구조를 변경하면 재배포 없이 사용자 화면 메뉴에 반영된다.
- 점검 모드 전환 시 일반사용자 접근이 차단되고 점검 안내가 노출된다.
- 일반사용자 역할 계정은 조회만 가능하며, 등록/수정/승인 등 권한 외 기능 요청은 거부된다.
- 로그인·개인정보 조회·권한 변경 행위 발생 시 각각 행위 유형·계정 ID·시각을 포함한 감사 로그가 생성된다.
- 접속 통계 화면에서 일별 사용자 추이와 기기별(PC/Mobile/Tablet) 비율이 조회된다.

**우선순위**: High (보안·감사 직결, HIGH SECURITY로 명시)

## 6. 비기능 요구사항

본 장은 비기능 요구사항을 클라우드/인프라, 데이터, 보안 영역으로 구분하여 정의한다. 각 요구사항은 NFR-ID로 식별하며, 측정 가능한 항목에 한해 수용 기준(검증 방법)을 명시한다. 우선순위는 필수(M, Mandatory) / 권장(R, Recommended)으로 구분한다.

### 6.1 클라우드/인프라 요구사항 (SLIDE 25~28)

#### 6.1.1 인프라 구축 전략

본 장은 사업 수행 과정에서 준수해야 할 기술적·법적·계약적 제약 조건을 정의한다. 각 제약사항은 제안서에 명시된 표준·법령·요건을 측정 가능한 형태로 정리하며, 제안서의 단정적 표현("100% 준수", "완벽 호환", "무결점" 등)은 검증 가능한 수용 기준으로 환원한다.

### 7.1 개발 언어 및 기술 스택 (SLIDE 39)

**설명**
목표시스템은 전자정부 표준 프레임워크(eGovFrame) 기반의 표준 기술 스택으로 구축하여 장기 유지보수성과 보안성을 확보한다. 프런트엔드는 웹 표준(HTML5/CSS3/ES6+) 기반의 Non-ActiveX 환경으로, 백엔드는 Java 및 eGovFrame 기반으로 구성한다.

**세부 요구사항**

| 계층 | 항목 | 요구 스펙 (원문값) |
|------|------|--------------------|
| Core Language | Java | 17 (LTS) — Record Class, Text Blocks, ZGC(Low Latency GC) 활용. Nashorn은 Deprecated 처리 |
| Standard Framework | eGovFrame | 4.2 (표준 프레임워크) |
| Application Framework | Spring Boot | 2.7 기반 |
| Data Mapper | MyBatis | 3.5 |
| Template Engine | Thymeleaf | SSR(Server Side Rendering), Spring MVC 통합, Natural Templating |
| Markup | HTML5 | 시맨틱 마크업, WAI-ARIA 기반 접근성 |
| Style/Layout | CSS3 | 반응형 웹(RWD), Flexbox/Grid Layout, Cross-Browser |
| Script | ES6+(JavaScript) | ECMAScript 최신 표준, Async/Await |
| 플러그인 정책 | Non-ActiveX | ActiveX 및 비표준 플러그인 미사용 |

**수용 기준**
- 빌드 산출물이 Java 17(LTS), eGovFrame 4.2, MyBatis 3.5 버전으로 구성됨이 빌드 메타데이터(예: 의존성 목록)로 확인된다.
- ActiveX 및 별도 클라이언트 설치형 플러그인 없이 지정 브라우저에서 전 기능이 동작함이 확인된다(Non-ActiveX 검증).
- 전자정부 표준 프레임워크 4.2 호환성 가이드의 점검 항목을 충족함을 진단 결과로 제시한다.

**우선순위**: 필수(Must)

> **비고 — 버전 정합성 확인 필요**: 원문(SLIDE 39)은 "eGovFrame 4.2"와 "Java 17(LTS)"를 함께 명시하면서 동시에 "Spring Boot 2.7 기반"으로 표기하고 있다. eGovFrame 4.x / Java 17 조합과 Spring Boot 2.7의 버전 정합성(Spring Boot 2.7은 통상 Java 8~Java 17 지원, eGovFrame 4.x 대응 Spring Boot 버전)은 착수 단계에서 발주처와 확정·검증할 필요가 있다. 본 PRD는 원문 표기값을 그대로 보존하되, 기술 스택 베이스라인 확정 시 해당 항목의 상호 호환성을 명시적으로 점검할 것을 권고한다.

### 7.2 웹 표준 및 웹 접근성 (SLIDE 40)

**설명**
국제 표준(W3C 웹 표준)과 한국형 웹 콘텐츠 접근성 지침(KWCAG 2.2)을 준수하여, 장애 유무 및 브라우저·OS 환경에 구애받지 않는 동등한 정보 접근성을 보장한다.

**세부 요구사항**
- **웹 표준**: W3C 웹 표준 준수, 시맨틱 태그 적용, HTML5/CSS3/ES6+ 표준 셋 적용, CSS/JS 경량화.
- **웹 접근성(KWCAG 2.2)**: 인식성·운용성·이해성·견고성 4대 원칙 준수.
- **보조기기 지원**: 스크린리더 호환, 키보드 접근성(키보드만으로 전 기능 조작).
- **대체 콘텐츠**: 이미지 대체 텍스트(alt) 제공, 멀티미디어 자막 제공.
- **멀티 브라우저**: Edge, Chrome, Safari, Whale 지원.
- **OS 무종속성**: Windows, Mac, Linux 환경 호환.
- **반응형 웹(RWD)**: CSS3 미디어 쿼리 기반, PC/Mobile/Tablet 해상도 대응.
- **내비게이션 구조**: GNB/LNB, 브레드크럼(Breadcrumbs), 전체 사이트맵 제공.

**수용 기준**
- KWCAG 2.2 4대 원칙 기준 항목에 대한 웹 접근성 진단 도구 검사 결과를 제출하고, 식별된 오류 항목이 개선 완료됨을 확인한다.
- Edge, Chrome, Safari, Whale 각 브라우저 및 Windows/Mac/Linux 각 OS에서 주요 화면이 정상 렌더링·동작함을 교차 검증 결과로 확인한다.
- 키보드 단독 조작 및 스크린리더 낭독으로 핵심 업무 흐름이 수행됨을 확인한다.
- 주요 페이지에 브레드크럼·사이트맵이 제공되고, 이미지/멀티미디어에 대체 텍스트·자막이 적용됨을 확인한다.

**우선순위**: 필수(Must) — 웹 접근성은 법적 의무 사항

### 7.3 법령 및 규정 변경 반영 (SLIDE 42)

**설명**
현행 법령 준수에 더하여 사업 기간 중 변경되는 법령·지침을 추적·반영하여, 향후 감사 및 운영 단계에서 컴플라이언스 이슈가 발생하지 않도록 한다.

**세부 요구사항**
- **대상 법령 준수**: 전자정부법, 개인정보 보호법, 국가정보화 기본법 등 관련 법령 준수.
- **법령 모니터링 프로세스**: 월 1회 법령 개정 추적, 법제도 모니터링 담당자(PMO) 지정.
- **변경 영향 반영**: 변경 사항을 아키텍처·개발 공정에 반영하고, 월간 보고 시 법령 변경 내역 및 영향도 분석을 보고.
- **기술 표준 적용**: eGovFrame 공통컴포넌트(240종) 활용, 정보시스템 구축·운영 지침 준수.
- **보안/클라우드 규정**: 국가정보보안기본지침 및 클라우드 보안인증(CSAP) 요건 충족.

**수용 기준**
- 법제도 모니터링 담당자(PMO)가 지정되고, 월 1회 주기의 법령 개정 추적 결과가 월간 보고서에 포함됨을 확인한다.
- 사업 기간 중 발생한 법령 변경 건에 대한 영향도 분석 및 반영 내역이 산출물로 기록됨을 확인한다.
- eGovFrame 공통컴포넌트(240종) 적용 현황이 산출물로 제시된다.
- CSAP 요건 충족 여부가 관련 산출물로 확인된다.

**우선순위**: 필수(Must)

### 7.4 계약 해지 조건 (SLIDE 41)

**설명**
발주처는 아래 귀책사유 발생 시 계약의 전부 또는 일부를 해지할 수 있다. 수행사는 이러한 사유의 사전 방지를 위해 품질 보증(QA) 조직을 별도 운영하고 품질 관리 체계를 가동한다.

**세부 요구사항 — 4대 해지 사유**

| 구분 | 해지 사유 |
|------|-----------|
| 01 수행 능력 부족 | 성과 기대가 곤란하거나 업무 수행 능력이 현저히 부재하다고 판단되는 경우 |
| 02 계약 미이행 | 정당한 사유 없이 계약 내용을 불이행하여 업무 추진에 상당한 지장을 초래한 경우 |
| 03 보안 및 불가피한 사유 | 비밀 준수 의무 위반 또는 중단 사유 발생 등 발주처가 불가피하다고 인정한 경우 |
| 04 과실 및 빈번한 오류 | 고의 또는 잦은 과실로 인한 오류 발생으로 업무 수행에 막대한 지장을 초래한 경우 |

**품질 보증 운영 체계**
- 전문 인력 투입 및 역량 검증 체계 가동.
- WBS 기반 진척 관리 및 정기 보고.
- 투입 인력 보안 서약 및 보안 감사 정례화.
- 3단계 품질 테스트(단위/통합/인수)를 통한 오류 최소화.

**수용 기준**
- QA 조직 구성 및 운영 계획이 산출물로 제시되고, 인력 역량 검증 결과가 확인된다.
- WBS 기반 진척률 보고가 정기적으로 이루어짐을 보고 산출물로 확인한다.
- 단위/통합/인수 3단계 테스트 수행 결과(테스트 케이스, 실행 결과)가 산출물로 제출된다.

**우선순위**: 필수(Must)

> **비고**: 상기 4대 해지 사유 및 "이의 제기 불가" 확약은 제안서의 수행사 측 확약 문구에 근거한다. 실제 계약 시 해지 사유·절차·효과는 발주처 계약 조건 및 관계 법령(국가계약법 등)에 따라 최종 확정되어야 한다.

### 7.5 특허권 및 저작권 (SLIDE 43)

**설명**
사업 산출물의 지식재산권은 공동 소유를 원칙으로 하며, SW 산출물의 외부 반출은 엄격한 보안 절차를 거쳐야 한다. 이를 통해 공동 활용 가치 창출과 정보 유출 방지를 동시에 달성한다.

**세부 요구사항 — 지식재산권 귀속 원칙**
- **공동 소유**: 발주처와 계약상대자가 지식재산권을 공동 소유한다(기획재정부 계약예규 제56조 준용).
- **상업적 활용**: 계약상대자는 산출물을 개작하여 상업적으로 활용할 수 있으며, 제3자 양도 시 발주처의 사전 승인을 필수로 한다.
- **특이사항(Note)**: 본 사업 SW는 타 기관과 공동 활용할 계획이 없음을 인지하고 사업을 수행한다.

**세부 요구사항 — SW 산출물 보안 반출 프로세스 (4단계)**

| 단계 | 주체 | 내용 |
|------|------|------|
| 01 반출 요청 | 공급자 | 지식재산권 활용 목적 명시 및 반출 신청서 제출 |
| 02 보안 검토 | 공급자 | 누출 금지 정보 완전 삭제 검증 및 대표자 명의 보안 확약서 작성 |
| 03 승인 심사 | 발주처 | 보안 규정 위배 여부 확인 및 반출 타당성 검토 후 승인 |
| 04 산출물 제공 | 관리자 | 승인된 산출물 반출(제3자 제공 시 별도 사전 승인 절차 수행) |

**수용 기준**
- 지식재산권 귀속(공동 소유) 및 상업적 활용·제3자 양도 시 사전 승인 조건이 계약 문서에 반영됨을 확인한다.
- SW 산출물 반출 시 4단계 절차(반출 요청 → 보안 검토 → 승인 심사 → 산출물 제공)에 따른 신청서·보안 확약서·승인 기록이 산출물로 확인된다.

**우선순위**: 필수(Must)

## 부록 B. 추적성 및 확인 필요 사항

본 PRD는 제안서 PPTX(SLIDE 1~43)에서 도출되었다. 작성 과정에서 식별된 가정·모호점·확인 필요 사항을 다음과 같이 정리한다. 착수(분석) 단계에서 발주처와 협의하여 확정할 것을 권고한다.

- **기술 스택 버전 정합성**: 원문(SLIDE 39)에 "eGovFrame 4.2 + Java 17(LTS) + Spring Boot 2.7"이 함께 명시되어 있다. Spring Boot 2.7과 Java 17 조합은 성립 가능하나, eGovFrame 4.x의 공식 대응 Spring Boot 버전 및 Java 17 지원 범위는 베이스라인 확정 시 상호 호환성을 명시적으로 검증해야 한다.
- **"1단계" 범위 경계 미정**: 본 사업이 "재구축(1단계)"로 명명되었으나, 제안서에는 단계 분리 기준(어떤 기능/시스템이 1단계 포함이고 무엇이 후속 단계인지)이 명시되지 않았다. 부록 A의 전체 메뉴 체계가 1단계 범위인지, 일부가 후속 단계인지 확인이 필요하다.
- **외부 연계 인터페이스 상세 미정**: NICE 평가정보, 가치평가, 운용사 ERP, 수탁기관 연계가 언급되었으나, 연계 방식·프로토콜·전문 규격·인증·연계 주기·SLA 등 인터페이스 상세 정의서가 존재하지 않는다. 별도 인터페이스 정의(I/F 설계) 산출물이 필요하다.
- **수용 기준 산식 미확정**: 고용창출 효과 산식(FR-5.4-02), IRR/ROI 계산식(FR-5.4-03, FR-5.7), 의무 투자 비율 임계치(FR-5.9) 등은 제안서에 예시값만 제시되어 정확한 산식·기준값은 업무 규칙 확정 후 반영해야 한다.
- **엑셀 일괄 업로드 부분 성공 정책**: 여러 기능(FR-5.1-08, FR-5.3-03, FR-5.8-08)에서 엑셀 일괄 업로드를 요구하나, 검증 실패 행 처리 정책(전체 롤백 vs 부분 성공 허용)이 미정이다.
- **마케팅 수사 → 측정 기준 환산**: 원문의 "100% 보장/완벽 호환/획기적/무결점/빈틈없는" 등 정성 표현은 본 PRD에서 측정 가능한 수용 기준(항목별 충족 입증, 불일치 0건, 진단도구 결과, RTO 4h/RPO 10m 등)으로 번역하였다. 각 환산 기준의 적정성은 발주처 검수 기준과 정합되는지 확인이 필요하다.
- **계약·지식재산권 조항의 법적 확정 필요**: 계약 해지 조건(7.4)과 지식재산권 귀속(7.5)은 제안서의 수행사 측 확약 문구에 근거한다. 실제 효력은 발주처 계약서 및 관계 법령(국가계약법, 기획재정부 계약예규 등)에 따라 최종 확정되어야 한다.
- **RBAC 3등급의 세부 권한 매트릭스 미정**: 슈퍼관리자/운용심사역/일반사용자 3등급이 정의되었으나, 부록 A의 약 130개 메뉴별 CRUD 권한 매핑(메뉴×등급 권한 매트릭스)은 별도 정의가 필요하다.
- **DR 이중화 수준 vs 서버 사이징의 정합성**: RTO 4h/RPO 10m(연 1회 DR 리허설)와 6.1.2 서버 사이징(운영 단일 구성으로 보임)이 DR 사이트 구성·이중화 범위 측면에서 정합되는지 확인이 필요하다(DB는 Master/Slave 이중화 명시되나 DR 사이트 규모는 미상).
- **단위 표기 일관성**: 원문에 "억원/억", "vCPU/vcore" 등 혼용 표기가 있다. 본 PRD는 원문 값을 보존하되, 산출물 확정 시 단위 표기 표준화가 필요하다.
